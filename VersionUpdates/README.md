# iCloush 智慧工厂 — 版本更新文档 SOP

## 概述

本文件夹用于归档 iCloush 智慧工厂管理系统每次大小版本更新的版本说明文档。所有版本更新（包括功能迭代、Bug修复、热修复补丁）均需在此目录下同步交付一份版本说明文档，作为项目交付物的一部分。

---

## 版本号体系

采用**三级语义化版本号**（Semantic Versioning）：**VX.Y.Z**

| 层级 | 格式 | 含义 | 触发场景 | 示例 |
|------|------|------|----------|------|
| 第一级 | **X**.0.0 | 大版本 / Phase 大迭代 | 系统架构升级、Phase 大版本发布、核心模块重构 | V4.0.0（Phase 4 基础架构） |
| 第二级 | X.**Y**.0 | 功能版本 / 新模块上线 | 新功能模块开发完成、现有模块功能扩展 | V4.1.0（成本明细账 + 车辆CRUD） |
| 第三级 | X.Y.**Z** | 补丁版本 / Bug 修复 | 不涉及新功能开发，仅修复错误、优化体验 | V4.5.1（批量修复6个Bug） |

**版本号递增规则**：

- 当进入新的 Phase 大迭代时，第一级版本号递增，后两级归零（如 V4.0.0 → V5.0.0）
- 当新功能模块上线时，第二级版本号递增，第三级归零（如 V4.1.0 → V4.2.0）
- 当仅修复 Bug 或优化细节时，第三级版本号递增（如 V4.5.0 → V4.5.1）

## 文件命名规范

```
iCloush智慧工厂——Phase{阶段号}_{功能模块简述}-V{X.Y.Z}.md
```

**命名示例**：

| 文件名 | 说明 |
|--------|------|
| `iCloush智慧工厂——Phase4_基础架构与核心模块-V4.0.0.md` | Phase 4 大版本发布 |
| `iCloush智慧工厂——Phase4.1_成本明细账与车辆管理-V4.1.0.md` | Phase 4.1 功能版本 |
| `iCloush智慧工厂——Phase4.1_服务端错误修复-V4.1.1.md` | Phase 4.1 补丁版本 |
| `iCloush智慧工厂——Phase4.5_发票中台终极重构-V4.5.0.md` | Phase 4.5 功能版本 |
| `iCloush智慧工厂——Phase4.5_批量Bug修复-V4.5.1.md` | Phase 4.5 补丁版本 |

**命名要求**：

1. 文件名中的 Phase 编号与 Git 提交中的 Phase 编号保持一致
2. 功能模块简述控制在 15 个汉字以内，概括本次更新的核心内容
3. 版本号严格遵循三级体系，不允许跳版本号
4. 文件格式统一为 Markdown（`.md`）

每份版本说明文档应包含以下结构：

```markdown

**版本号**: VX.Y.Z
**发布日期**: YYYY-MM-DD
**Git 提交**: `哈希值`
**版本类型**: 大版本 / 功能版本 / 补丁版本

---

## 更新概述

（一段话概括本次更新的核心内容和目的）

## 更新内容

### 新增功能（功能版本必填）
（表格列出新增功能清单）

### Bug 修复（补丁版本必填）
（表格列出修复的Bug清单，含优先级、根因、修复方案）

### 优化改进（可选）
（列出性能优化、体验改进等）

## 涉及文件

（列出本次修改的文件清单）

## 部署步骤

（列出部署/更新操作步骤）

## 已知问题

（列出本版本已知但未修复的问题，供下个版本跟进）

## 下一版本规划

（简要说明下个版本的计划方向）
```

## 交付流程 SOP

1. **代码提交**：完成开发/修复后，按规范提交 Git commit
2. **撰写文档**：在 `VersionUpdates/` 目录下创建版本说明文档
3. **版本号确认**：根据更新类型确定版本号层级（大版本/功能版本/补丁）
4. **一并推送**：将代码和版本文档一起 `git push` 到 GitHub
5. **通知确认**：向项目负责人交付版本说明，确认部署计划
